Configuration Guide

Complete configuration guide for Agentic Local Brain, covering model setup, provider options, and all config.yaml fields.

Configuration File

Location: ~/.localbrain/config.yaml

Created automatically by localbrain init

Full Example Configuration

# Data storage directory
data_dir: ~/.knowledge-base

# LLM configuration (for tagging, RAG, entity extraction, topic labels)
llm:
  provider: litellm            # litellm (recommended) | dashscope | openai_compatible
  model: dashscope/qwen-plus   # provider/model format for litellm
  api_key: ${DASHSCOPE_API_KEY} # env var reference or raw key
  base_url:                     # optional, for custom endpoints

# Embedding model (for semantic search, document similarity)
embedding:
  provider: litellm             # litellm (recommended) | dashscope | openai
  model: openai/text-embedding-v4
  api_key: ${DASHSCOPE_API_KEY}
  base_url: https://dashscope.aliyuncs.com/compatible-mode/v1

# Knowledge Mining
knowledge_mining:
  entity_extraction:
    enabled: true
  topic_clustering:
    min_cluster_size: 5         # minimum docs to form a topic

# Text chunking
chunking:
  max_chunk_size: 1000          # max chars per chunk
  chunk_overlap: 100            # overlap between chunks

# Vector storage
storage:
  type: chroma
  persist_directory: ~/.knowledge-base/db/chroma

# Logging
logging:
  level: INFO                   # DEBUG | INFO | WARNING | ERROR
  log_dir: ""                   # empty = ~/.localbrain/logs/
  max_bytes: 10485760           # 10MB
  backup_count: 5

# RAG Configuration (v0.7)
query:
  rag:
    top_k: 5                    # Number of documents to retrieve
    temperature: 0.3             # LLM generation temperature
    max_tokens: 1000             # Max tokens for LLM response
    context_budget: 4000         # Context token budget
    context_format: hierarchical # hierarchical | flat
    reranking:
      enabled: true              # Enable LLM reranking
      top_n_candidates: 20       # Candidates for reranking
      weight_retrieval: 0.4      # Retrieval score weight
      weight_rerank: 0.6         # Rerank score weight
    conversation:
      max_turns: 20              # Max turns per session
      session_timeout_minutes: 30  # Session timeout
      history_turns_in_context: 5  # History turns in context
    templates:
      default: general           # general, technical, academic, creative
  pipeline:
    top_k: 10                    # Retrieval top-k
    rerank_top_k: 5              # Reranked top-k

# Wiki Configuration (v0.7)
wiki:
  enabled: true                        # Enable wiki generation
  max_source_tokens_per_topic: 8000    # Max tokens from source docs
  entity_card_threshold: 3             # Min topic appearances for entity cards
  temperature: 0.3                     # LLM temperature for compilation
  model: null                          # null = use default LLM
  max_article_words: 3000              # Target max words per article
  max_subcategories: 5                 # Max subcategories per topic

Model Configuration

Configure AI models for embedding and LLM services. This guide covers all supported providers and their configuration options.

Embedding Models

Embedding models convert text into numerical vectors (embeddings). These vectors are used for:

  • Semantic Search: Find documents with similar meaning, not just matching keywords
  • Document Similarity: Discover related documents in your knowledge base
  • RAG Context Retrieval: Retrieve relevant context for question answering

Embeddings are generated during document collection and stored in ChromaDB for fast similarity search.

LLM (Large Language Models)

LLMs provide intelligent text processing capabilities:

  • Auto-tagging: Automatically extract relevant tags from document content during collection
  • Summarization: Generate document summaries
  • Entity Extraction: Identify and extract named entities (people, organizations, concepts) during knowledge mining
  • Knowledge Graph Construction: Build relationships between entities and documents
  • Topic Labeling: Generate human-readable labels for discovered topic clusters
  • RAG Q&A: Answer questions using retrieved document context

Graceful Degradation

The system works progressively based on available services:

  • Without Embedding: Collection works, keyword search works, semantic search unavailable, RAG falls back to keyword search
  • Without LLM: Collection works with built-in tag extraction, knowledge mining and knowledge graph construction require LLM
  • Without Both: Basic collection and keyword search only

The recommended option for users in China. Provides stable API access to Qwen models.

llm:

Ollama (Local Models)

Run models locally using Ollama. Great for privacy and zero API costs.

⚠️ Performance Note

Local models are suitable for lightweight use. For large-scale processing or batch operations, cloud models provide better performance.

Installation

  1. Install Ollama from ollama.com
  2. Pull the models you want to use:
ollama pull qwen2.5:7b        # For LLM tasks
ollama pull nomic-embed-text  # For embeddings

LLM Configuration

llm:
  provider: litellm
  model: openai/qwen2.5:7b
  api_key: ollama
  base_url: http://localhost:11434

Embedding Configuration

embedding:
  provider: litellm
  model: openai/nomic-embed-text
  api_key: ollama
  base_url: http://localhost:11434

Recommended Configurations

Lightweight / Personal Use

For individual users with smaller knowledge bases:

  • Provider: Ollama (local)
  • Cost: Zero (runs on your hardware)
  • Best for: Privacy-conscious users, small collections, offline usage

Production / Large-Scale

For teams or large knowledge bases:

  • Provider: DashScope or OpenAI (cloud)
  • Cost: Pay per usage
  • Best for: Batch processing, fast embedding generation, reliable API access

Knowledge Mining Configuration

Knowledge Mining automatically extracts entities, discovers document relationships, and clusters topics.

FieldTypeDefaultDescription
knowledge_mining.entity_extraction.enabledbooltrueEnable LLM-based entity extraction on new documents
knowledge_mining.topic_clustering.min_cluster_sizeint5Minimum documents required to form a topic cluster (HDBSCAN parameter)

Behavior

  • Auto-trigger on new documents: When a document is added, entity extraction, embedding generation, and relation discovery run automatically in a background thread (~14s per document)
  • Topic clustering: Global operation, not triggered per-document. Use the "Rebuild Topics" button on the Topics page or localbrain topics rebuild
  • Full pipeline CLI: localbrain mine run (incremental) or localbrain mine run --full (rebuild all)
  • Concurrency: localbrain mine run --workers 5 for parallel entity extraction

模型配置

配置用于嵌入和 LLM 服务的 AI 模型。本指南涵盖所有支持的提供商及其配置选项。

嵌入模型 (Embedding Models)

嵌入模型将文本转换为数值向量（嵌入）。这些向量用于：

  • 语义搜索：查找含义相似的文档，而不仅仅是关键词匹配
  • 文档相似度：在知识库中发现相关文档
  • RAG 上下文检索：为问答检索相关上下文

嵌入在文档收集期间生成，并存储在 ChromaDB 中以实现快速相似度搜索。

大语言模型 (LLM)

LLM 提供智能文本处理能力：

  • 自动打标签：在收集期间自动从文档内容中提取相关标签
  • 摘要生成：生成文档摘要
  • 实体抽取：识别并提取命名实体（人物、组织、概念）
  • 知识图谱构建：建立实体和文档之间的关系
  • 主题标签：为发现的主题聚类生成人类可读的标签
  • RAG 问答：使用检索到的文档上下文回答问题

优雅降级

系统根据可用服务逐步工作：

  • 无嵌入服务：收集工作正常，关键词搜索正常，语义搜索不可用，RAG 回退到关键词搜索
  • 无 LLM 服务：收集工作正常（使用内置标签提取），知识挖掘和知识图谱构建需要 LLM
  • 两者都没有：仅基本收集和关键词搜索

运行机制

  • 新文档自动触发：文档入库后，实体抽取、Embedding 生成、关系发现会在后台线程自动执行（每篇约 14 秒）
  • 主题聚类：全局操作，不按单篇触发。在主题页面点击"重建主题聚类"按钮，或执行 localbrain topics rebuild
  • 全量管道 CLI：localbrain mine run（增量）或 localbrain mine run --full（全量重建）
  • 并发加速：localbrain mine run --workers 5 并行实体抽取
