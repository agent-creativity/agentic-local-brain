## 1. 概述

### 1.1 产品定位

知识收集(v0.5)解决"存"和"找"的问题——用户主动存入、主动搜索。知识挖掘(v0.6)解决"发现"的问题——系统主动揭示隐藏的结构和关联,将碎片化知识转化为体系化的认知地图。

核心价值: 用户不知道自己不知道什么,系统帮助发现。

### 1.2 功能范围

| 特性 | 优先级 | 版本 |
|------|--------|------|
| 自动知识图谱构建 | P0 | v0.6.0 |
| 跨文档关系发现 | P0 | v0.6.0 |
| 主题聚类和趋势分析 | P1 | v0.6.0 |
| 基于阅读模式的智能推荐 | P1 | v0.6.0 |
| 时间线视图 | P2 | v0.7 |
| AI 领域综述生成 | P2 | v0.7 |
| 知识缺口分析 | P2 | v0.7 |

### 1.3 实现顺序

图谱 -> 跨文档关系 -> 主题聚类 -> 智能推荐

依赖关系: 跨文档关系依赖图谱的共享实体数据; 推荐依赖主题聚类和阅读历史。

### 1.4 Tags vs Topics 定位澄清

| 维度 | Tags (现有) | Topics (v0.6 新增) |
|------|------------|-------------------|
| 定位 | 用户态标签 | 系统态聚类 |
| 粒度 | 细 (单个关键词) | 粗 (知识领域) |
| 生成 | LLM 逐篇提取 | embedding 全局聚类 |
| 可编辑 | 是 | 否 (自动生成) |
| 用途 | 搜索过滤、文档标注 | 全局可视化、趋势分析 |
| 存储 | tags 表 (独立) | topic_clusters 表 (独立) |
| 前端位置 | 文档卡片筛选器 | 图谱/气泡图全景导航 |
| 可验证性 | 离散字符串, 易测试 | 无 ground truth, 测稳定性 |

**原则**: 两者独立, 不互相污染。Topics 不写入 tags 表。

**预留协同方向(不在 v0.6 实现)**:
- 主题聚类参考高频 tags 生成更准确的标签名
- 聚类结果反向评估 tag 提取一致性
- 如需关联, 通过 `source: cluster` metadata 区分

## 2. 场景与展现

### 2.1 知识全景图 (Knowledge Map)

**场景**: 用户收集了大量文档,但不知道自己的知识结构全貌。

**展现方式**:
- 力导向图谱 (ECharts graph): 节点=文档/实体,边=关系,聚类着色,可缩放/点击
- 主题气泡图 (ECharts scatter): 每个气泡=一个主题,大小=文档数量,颜色深浅=近期活跃度

### 2.2 跨文档洞察 (Cross-Document Insights)

**场景**: 用户分别收藏了关于不同主题的文档,未意识到它们之间存在交叉关联。

**展现方式**:
- 文档详情页增加"相关文档"侧边栏,展示关联文档和关联理由
- 图谱视图中点击节点,高亮所有关联路径

### 2.3 知识趋势 (Knowledge Timeline)

**场景**: 用户想了解自己的学习轨迹和关注点变化。

**展现方式**:
- 趋势折线图: 各主题的文档增长趋势 (v0.6)
- 时间线视图: 横轴时间,纵轴主题,节点=文档 (v0.7)

### 2.4 智能推荐

**场景**: 用户在研究某个主题,不知道库中还有哪些相关未读内容。

**展现方式**:
- Web 首页推荐卡片: 每次显示 3-5 条推荐,附理由
- 文档详情页底部"延伸阅读"

## 4. 特性 1: 自动知识图谱构建

### 4.1 数据模型

```sql
-- 实体表
CREATE TABLE entities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,               -- 实体名称(normalized: lowercase + strip)
    display_name TEXT NOT NULL,       -- 原始显示名称
    type TEXT NOT NULL,               -- person / concept / tool / project / organization
    description TEXT,                 -- LLM 生成的实体描述
    mention_count INTEGER DEFAULT 1,  -- 被提及总次数
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(name, type)
);

CREATE INDEX idx_entities_type ON entities(type);
CREATE INDEX idx_entities_name ON entities(name);

-- 实体-文档关联表
CREATE TABLE entity_mentions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_id INTEGER NOT NULL,
    knowledge_id INTEGER NOT NULL,
    context TEXT,                     -- 提及上下文(原文片段)
    FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    UNIQUE(entity_id, knowledge_id)
);

CREATE INDEX idx_entity_mentions_knowledge ON entity_mentions(knowledge_id);

-- 实体关系表
CREATE TABLE entity_relations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_entity_id INTEGER NOT NULL,
    target_entity_id INTEGER NOT NULL,
    relation_type TEXT NOT NULL,      -- uses / belongs_to / related_to / depends_on / created_by
    weight REAL DEFAULT 1.0,         -- 自动按 source 数量计算
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (source_entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    FOREIGN KEY (target_entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    UNIQUE(source_entity_id, target_entity_id, relation_type)
);

-- 关系来源表(多对多: 一条关系可能在多篇文档中被提及)
CREATE TABLE entity_relation_sources (
    relation_id INTEGER NOT NULL,
    knowledge_id INTEGER NOT NULL,
    context TEXT,                     -- 关系在该文档中的上下文
    PRIMARY KEY (relation_id, knowledge_id),
    FOREIGN KEY (relation_id) REFERENCES entity_relations(id) ON DELETE CASCADE,
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE
);
```

### 4.2 实体类型 Schema

| 类型 | 标识 | 示例 |
|------|------|------|
| 人物 | person | Yann LeCun, 张三 |
| 概念/术语 | concept | Transformer, 反向传播, 微服务 |
| 工具/技术 | tool | PyTorch, Docker, LiteLLM |
| 项目/产品 | project | GPT-4, Kubernetes, localbrain |
| 组织 | organization | OpenAI, Google |

### 4.3 关系类型 Schema

| 关系 | 标识 | 方向 | 示例 |
|------|------|------|------|
| 使用 | uses | A uses B | PyTorch uses CUDA |
| 属于 | belongs_to | A belongs_to B | BERT belongs_to NLP |
| 关联 | related_to | A related_to B | 通用双向关联 |
| 依赖 | depends_on | A depends_on B | LiteLLM depends_on httpx |
| 创建 | created_by | A created_by B | Transformer created_by Google |

### 4.4 EntityExtractor 实现

```python
class EntityExtractor(BaseProcessor):
    """
    实体和关系抽取处理器。
    在 processing pipeline 中与 TagExtractor 并行执行。

    设计原则(Alice 建议):
    - extract() 负责 LLM 调用和解析,返回 Python 对象,可独立 mock 测试
    - save() 负责写入 SQLite,可用 in-memory SQLite 测试
    """

    ENTITY_TYPES = {"person", "concept", "tool", "project", "organization"}
    RELATION_TYPES = {"uses", "belongs_to", "related_to", "depends_on", "created_by"}

    def extract(self, title: str, content: str) -> dict:
        """
        LLM 抽取实体和关系。

        返回:
            {
                "entities": [{"name": "...", "type": "...", "description": "..."}],
                "relations": [{"source": "...", "target": "...", "type": "...", "context": "..."}]
            }

        约束:
            - 每篇文档最多 10 个实体、10 条关系
            - temperature = 0.1
            - 输出格式: JSON, 严格遵循 schema
        """
        pass

    def save(self, extracted: dict, knowledge_id: int) -> None:
        """
        将抽取结果写入 SQLite。

        - 实体写入前 normalize: name.lower().strip()
        - 同名同类型实体去重, mention_count += 1
        - 关系去重, 新增 source 到 entity_relation_sources
        - weight 按 source 数量自动更新
        """
        pass

    def process(self, title: str, content: str, knowledge_id: int) -> ProcessResult:
        """完整流程: extract -> save"""
        extracted = self.extract(title, content)
        self.save(extracted, knowledge_id)
        return ProcessResult(success=True, data=extracted)
```

### 4.5 实体名称规范化策略

写入前执行:
1. `name.lower().strip()` -- 基础规范化
2. 去除多余空格 -- `re.sub(r'\s+', ' ', name)`
3. 存储 `display_name` 保留原始形式(首次出现的形式)

v0.6 不做 LLM entity linking(成本高), 后续版本可选增加。

### 4.6 图谱查询

N 跳关系查询使用 SQLite 递归 CTE:

```sql
WITH RECURSIVE related(entity_id, depth) AS (
    SELECT target_entity_id, 1
    FROM entity_relations
    WHERE source_entity_id = :start_id
    UNION ALL
    SELECT er.target_entity_id, r.depth + 1
    FROM entity_relations er
    JOIN related r ON er.source_entity_id = r.entity_id
    WHERE r.depth < :max_depth
)
SELECT DISTINCT e.* FROM related r
JOIN entities e ON e.id = r.entity_id;
```

## 5. 特性 2: 跨文档关系发现

### 5.1 关系发现方式

**A. 基于 Embedding 相似度:**
- 文档级 embedding: 使用 title + 摘要(前 500 字)单独生成(Tom 建议, 语义更集中)
- 计算文档对余弦相似度
- 阈值 >= 0.75 建立关系(可配置)

**B. 基于共享实体:**
- 两篇文档提到相同实体 -> 自动关联
- 共享实体数量作为关系权重

### 5.2 数据模型

```sql
-- 文档级 embedding 缓存
CREATE TABLE document_embeddings (
    knowledge_id INTEGER PRIMARY KEY,
    embedding BLOB NOT NULL,          -- 文档级 embedding(title+摘要)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE
);

-- 文档间关系表
CREATE TABLE document_relations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_knowledge_id INTEGER NOT NULL,
    target_knowledge_id INTEGER NOT NULL,
    relation_type TEXT NOT NULL,       -- embedding_similarity / shared_entity
    score REAL NOT NULL,              -- 相似度分数 或 共享实体数
    shared_entities TEXT,             -- JSON: 共享实体 ID 列表(仅 shared_entity 类型)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (source_knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    FOREIGN KEY (target_knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    UNIQUE(source_knowledge_id, target_knowledge_id, relation_type)
);

CREATE INDEX idx_doc_relations_source ON document_relations(source_knowledge_id);
CREATE INDEX idx_doc_relations_target ON document_relations(target_knowledge_id);
```

### 5.3 计算时机

- **增量**: 文档入库时, 新文档 vs 已有文档计算相似度和共享实体
- **批量**: CLI 命令 `kb graph rebuild` 全量重算
- **性能**: 增量计算时仅与最近 1000 篇文档比较(避免 O(n^2) 全量扫描)

## 6. 特性 3: 主题聚类和趋势分析

### 6.1 聚类方案

- **算法**: HDBSCAN (`min_cluster_size=5`, 可通过 config 调整)
- **输入**: 所有文档的文档级 embedding(特性 2 中生成的 title+摘要 embedding)
- **输出**: 每个 cluster 分配一个 LLM 生成的主题标签和描述
- **噪声处理**: HDBSCAN 自动标记不属于任何主题的文档为噪声点(label=-1)

### 6.2 数据模型

```sql
CREATE TABLE topic_clusters (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    label TEXT NOT NULL,              -- LLM 生成的主题名
    description TEXT,                 -- 主题描述
    document_count INTEGER DEFAULT 0,
    centroid_embedding BLOB,          -- 聚类中心 embedding
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE knowledge_topics (
    knowledge_id INTEGER NOT NULL,
    cluster_id INTEGER NOT NULL,
    confidence REAL DEFAULT 1.0,      -- 归属置信度
    PRIMARY KEY (knowledge_id, cluster_id),
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    FOREIGN KEY (cluster_id) REFERENCES topic_clusters(id) ON DELETE CASCADE
);
```

### 6.3 计算策略

- **全量重算**: `kb topics rebuild` -- 重新聚类所有文档, 重新生成主题标签
- **增量分类**: 新文档入库时, 计算与已有 cluster centroid 的余弦距离, 分配到最近的 cluster(距离超阈值则标记为未分类, 等下次全量重算)
- **Centroid 更新**: 增量分配时使用 running average 更新 centroid(Tom 建议)
  ```
  new_centroid = (old_centroid * n + new_doc_embedding) / (n + 1)
  ```
- **重算触发条件**: 文档数增长 20% 或手动触发

### 6.4 主题标签生成

每个 cluster 取其中 5 篇代表性文档(距 centroid 最近)的标题, 用 LLM 生成:
- 主题名称(2-6 字)
- 主题描述(一句话)

## 7. 特性 4: 基于阅读模式的智能推荐

### 7.1 数据模型

```sql
CREATE TABLE reading_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    knowledge_id INTEGER,             -- 访问的文档(纯搜索时为 NULL)
    query TEXT,                       -- 搜索查询词
    action_type TEXT NOT NULL,        -- search / view / rag_query
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- 预留字段(v0.7)
    duration_seconds INTEGER,
    interaction_type TEXT,
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE SET NULL
);

CREATE INDEX idx_reading_history_time ON reading_history(created_at DESC);
CREATE INDEX idx_reading_history_knowledge ON reading_history(knowledge_id);
```

### 7.2 数据采集

通过 Web API middleware 记录(Alice 建议, 不侵入业务逻辑):
- `GET /api/knowledge/{id}` -> 记录 view
- `GET /api/search` -> 记录 search
- `POST /api/rag` -> 记录 rag_query

### 7.3 推荐算法 (v0.6 简单版)

1. 取最近 N 次(默认 20)阅读/查询的文档 embedding
2. 计算时间衰减加权均值: `weight = exp(-lambda * days_ago)`
3. 在文档级 embedding 中找最相似的、用户近期未访问的文档
4. 可选: 同主题 cluster 内的文档额外加分(+0.1)
5. 返回 top-K(默认 5)

### 7.4 推荐理由生成

基于关联方式生成自然语言理由:
- 基于 embedding 相似: "与你最近阅读的《XXX》内容相关"
- 基于共享实体: "与《XXX》共同涉及 PyTorch、CUDA 等技术"
- 基于同主题: "同属于「深度学习」主题下的文档"

| 模块 | 测试方法 | 要点 |

- `tests/fixtures/embeddings/` -- 固定 embedding fixture, 覆盖 2-3 个明显不同主题

- 每个特性完成后全量回归(防止 schema 变更影响现有功能)
- 图谱 API 端到端测试(写入 -> 查询 -> 验证返回格式)

### 11.4 Schema 验证优先

LLM 输出不确定性大, 测试重点放在:
- 输出结构是否符合预定义 schema
- 实体类型/关系类型是否在白名单内
- 不验证具体内容正确性(LLM 输出非确定性)

  # 跨文档关系

    algorithm: hdbscan

  # 推荐
  recommendation:
    recent_history_count: 20          # 参考最近 N 条阅读记录

## 14. 技术决策记录

| 决策 | 选项 | 结论 | 原因 |
|------|------|------|------|
| 图谱存储 | SQLite vs Neo4j | SQLite | 个人知识库规模不大, 避免运维成本; 复杂图算法用 NetworkX 内存计算 |
| 文档级 embedding | chunk 均值 vs title+摘要 | title+摘要 | 语义更集中, 一次 embedding 调用成本低 (Tom 建议) |
| 聚类算法 | KMeans vs HDBSCAN | HDBSCAN | 自动发现主题数量, 能标记噪声点 (Tom 建议) |
| 实体去重 | normalize vs LLM linking | normalize (v0.6) | lowercase+strip 成本低; LLM linking 效果好但成本高, 留到后续版本 |
| 关系模型 | 单表 vs 多对多 | 多对多 | 一条关系可被多文档提及, weight 按 source 数量计算 (Tom 建议) |
| 前端图谱库 | D3.js vs ECharts | ECharts | API 简洁, 中文文档好, 集成成本低 (Tom 建议) |
| 推荐复杂度 | 协同过滤 vs embedding 相似 | embedding 相似 (v0.6) | v0.6 先做简单版, 预留扩展字段 |
| reading_history 写入 | 业务层 vs middleware | middleware | 不侵入业务逻辑, 测试更干净 (Alice 建议) |
| EntityExtractor 架构 | 单方法 vs extract+save | extract+save | LLM 测试和存储测试分离, 不相互污染 (Alice 建议) |
